# Reading the LAN8680 front-end over I²C (`lan866x-lan8680`)

> **TL;DR** The LAN8680 T1S front-end (the board's **System Basis Chip**) exposes
> its control/status registers over an I²C **servicing** interface at the fixed
> address **0x40**. `lan866x-lan8680` finds the housekeeping bus, reads the chip's
> identity (PHY_ID1/2) and, with `--status`, decodes its supply/temperature
> warnings — **read-only**.

> 📄 **Source:** [../lan8680.c](../lan8680.c) — core: [../src/rcp.c](../src/rcp.c) /
> [../src/rcp.h](../src/rcp.h). Related I²C example: [I2CDEMO.md](I2CDEMO.md).
> Demo index: [DEMOS.md](DEMOS.md).

> ⚠️ **READ-ONLY by design.** The LAN8680 is the board's **power supply (LDOs),
> watchdog, reset and voltage/temperature monitor**. Reads are safe; **writes can
> reset or power down the whole board**. Writing is intentionally not implemented
> (and is non-trivial: a custom bit-enable + XOR-parity protocol, plus configuration
> protection via the CFGPRTCTL key sequence A505h/055Ah + WREN).

---

## 1. Two chips, two management interfaces

The LAN8680 has **two** register spaces with **different** access methods (from the
LAN8680 datasheet DS60001831B):

| Block | Interface | How |
|---|---|---|
| PMD **transceiver** (the T1S PHY) | **MDIO / SMI** (Clause 22 MIIM) | not over I²C |
| **System Basis** (power, watchdog, monitors, DIO) | **I²C** | what this tool uses |

So "access the LAN8680 over I²C" means the **System Basis** side. The LAN8661 MCU is
the I²C **master**; the LAN8680 is **slave-only** at **0x40** (it can never be the
manager).

## 2. The I²C protocol (System Basis registers)

- **Address:** static 7-bit **0x40**.
- **Registers are 16-bit**, transmitted **MSB first**.
- **Read** = write the 1-byte register address → repeated-START → read 2 bytes:
  `value = (byte0 << 8) | byte1`. This maps exactly onto RCP `WriteAndReadI2C`
  (`0x1208`): `WriteData=[reg]`, `ReadDataLength=2`.
  - (Contrast: the VCNL4200 in [I2CDEMO.md](I2CDEMO.md) is **LSB first**.)
- **Write** (not used here) is a custom frame: register address, a 16-bit *bit-enable*
  mask, the 16-bit data, then a 16-bit *bit-enable XOR data* parity word — and
  config-protected fields additionally need the CFGPRTCTL unlock keys + WREN.

### Identity & status registers

| Reg | Addr | Meaning |
|---|---|---|
| `PHY_ID1` | 0x40 | OUI low → `0x0007` (Microchip OUI 00800Fh) |
| `PHY_ID2` | 0x41 | OUI high + **MODEL[5:0]=100000** (LAN8680) + REV — reset `0xC200` |
| `STS0` | 0x43 | SBC Status 0: over-temp / battery / VS / VSEN / VUC warnings |
| `STS1` | 0x44 | SBC Status 1 |

The tool confirms a LAN8680 by reading `PHY_ID2` and checking `MODEL == 0x20`.

## 3. Finding the bus (why it auto-probes)

The LAN8680 sits on the **housekeeping I²C** (`SDA-R/SCL-R`), which is one of the
MCU's four SERCOMs (configurable in firmware; **disabled/`None`** in the shipped
Lighting build). A scan on the Click application I²C (SER2 = PA08/PA09) shows only
the VCNL4200 (0x51) — **not** the LAN8680 — so the housekeeping bus is a *different*
SERCOM. `lan866x-lan8680` therefore **auto-probes** the SERCOM I²C pin pairs until
0x40 answers with a valid LAN8680 ID:

| SERCOM | SDA/SCL (PA) | normally used for |
|---|---|---|
| SER2 | 8 / 9 | Click application I²C (proximity) |
| SER0 | 0 / 1 | WS2812 Display 0 |
| SER1 | 4 / 5 | WS2812 Display 1 |
| SER3 | 12 / 13 | SPI thumbstick |

> ⚠️ Probing `OpenI2C`s on SERCOMs that may currently drive the displays (SER0/SER1)
> or the SPI thumbstick (SER3); this can briefly glitch them. The pins are released
> afterwards. If you already know the bus, pass `--sda/--scl` to probe only that one.

## 4. Run it

```bat
release\lan866x-lan8680.exe --ip 192.168.0.54
release\lan866x-lan8680.exe --ip 192.168.0.54 --status
release\lan866x-lan8680.exe --sda 4 --scl 5            REM probe only SER1
release\lan866x-lan8680.exe --reg 0x44                 REM read one register (STS1)
```

Expected (when the LAN8680 is reached):

```
Probing SERCOM I2C buses for the LAN8680 @ 0x40 ...
  SER2 PA08/PA09 ... no
  SER0 PA00/PA01 ... FOUND

LAN8680 found on SER0 (SDA=PA00 SCL=PA01, addr 0x40)
  PHY_ID1 (0x40) = 0x0007
  PHY_ID2 (0x41) = 0xC200   MODEL=0x20  REV=0
  -> LAN8680 PMD Transceiver System Basis Chip confirmed.
```

| Option | Meaning |
|---|---|
| `--sda <n>` / `--scl <n>` | probe only this one bus (skip auto-probe) |
| `--status` | also read & decode `STS0` (0x43) warnings |
| `--reg <r>` | read one 16-bit register and print it (hex ok) |
| `--ip` / `--ep` | target endpoint |

## 5. Caveats & troubleshooting

| Symptom | Likely cause / fix |
|---|---|
| `LAN8680 not found on any SERCOM` | housekeeping bus on a non-default routing, or not wired to a probeable SERCOM in this build; try explicit `--sda/--scl`, or check the schematic's `SDA-R/SCL-R` net. |
| a display goes dark after probing | expected — probing reconfigured its SERCOM as I²C; re-run the display source (or `clickdemo`) to restore it. |
| `OpenI2C failed` on a bus | that SERCOM is busy/!routable; the probe moves on. |
| want to read DIO0/DIO1 (the buttons) | the front-end button inputs are in the SBC registers (CFG1/STS0 DIO bits) — read `--reg` per the datasheet; see [TOOLS.md §2.7](../TOOLS.md#27-on-board-leds--buttons). |
